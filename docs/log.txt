    PUT-annotations route now calls jobs.schedule(slide_id) after saving; the worker
    coalesces rapid edits and runs extract_patches -> embed_slide -> train (global).
    Graceful states: need_2_classes/no_data -> "waiting", EmbedderError -> "error",
    success -> "ready". Exposed via GET /api/learn/status.

    and a live similarity request don't hit torch/GPU concurrently.
  * Similarity is now annotation-based: models.SimilarityRegion carries the full W3C

    MEAN embedding of the annotation's tissue tiles (_annotation_reference_vector,

    empty_annotation -> 400.
  * Frontend: removed the Extract/Train top-bar buttons and the patch-montage +

2026-07-08 — Predict view: region boundaries instead of a heatmap.
  "Predict view" now outlines the regions the trained head calls the selected
  class with >= 80% probability, rather than painting a blue->red heatmap. Outline
  only, tile-grid (blocky) edges, fixed 80% threshold. "Find similar" (annotation
  similarity) is unchanged and still uses the heatmap.
  * backend/inference.py: added PREDICT_MIN_CONFIDENCE = 0.8; predict() now returns
    boundary SEGMENTS ([[x1,y1,x2,y2],...] in region-local px, via new
    _boundary_segments: an inside tile (score>=thr) emits each edge bordering a
    non-inside neighbor) plus n_above + min_confidence, and no longer writes a

2026-07-13 — Otsu tissue mask (now the default; the fixed-saturation mask stays available).
  Not the "one-line TISSUE_FN swap" I'd promised, and the reason is the interesting part:
  Otsu ALWAYS finds a split. Run it per tile and a tile of blank glass gets its sensor noise
  divided into "dark" and "light", reporting half the background as tissue. The histogram it
  reads must actually contain both populations — so the threshold has to be derived once from
  a view of the WHOLE slide. That makes the tissue gate per-slide, not a pure per-tile
  function, so the interface changed.
  * patches.py — `is_tissue(img)` / `TISSUE_FN` / `tissue_mask_id()` replaced by
    `tissue_gate(slide) -> TissueGate`: a callable `gate(tile) -> bool` carrying its cutoff and
    an `id`. Two gates, chosen by SLIDEPROBE_TISSUE ("otsu" default | "hsv"):
      - hsv:  the old fixed cutoff (saturation >= 25). Predictable, stain-blind.
      - otsu: cutoff LEARNED from the slide's own saturation histogram, read from a low-res
        overview (`_overview`, longest side 2048, taken from the BOUNDED region — not
        slide.get_thumbnail, which would include the scanner's dead margin and skew it).
        `_otsu()` is classic between-class-variance maximization over the 256-bin histogram,
        pure Python — no numpy/scikit-image. Clamped to [8, 80]: if a slide really is nearly
        all tissue or nearly all background, Otsu is splitting noise and its threshold must
        not be trusted.
    Both gates then keep a tile whose saturated fraction >= MIN_TISSUE_FRACTION (0.30).
  * features.py — caches the gate per (slide, mask name) since Otsu costs a low-res read;
    the sweep builds it ONCE and applies it to every tile (still exactly one call site). Each
    bank now records `tissue_mask` (e.g. "otsu-sat62-frac0.3") + `tissue_threshold`, validated
    on load — LAST, after the cheap checks, so we don't pay for an overview read to reject a
    bank the grid/model check already killed.
  * classifier.py metadata records `tissue_mask` as the mask NAME ("otsu"/"hsv").
  Measured on the real slides (thresholds Otsu derived): CMU-1-Small-Region 62, CMU-1 45,
  CMU-3 42, JP2K 52 — all well above the fixed 25, overview read ~0.2s.
  HONEST RESULT: on these well-stained H&E slides Otsu changes almost nothing. It is a strict
  SUBSET of the hsv mask (otsu-only = 0 tiles), trimming only edge tiles: CMU-1 5,778 vs 5,798
  tissue tiles (20 fewer of 29,930); CMU-1-Small-Region 48 vs 51. That is reassuring rather
  than disappointing — tissue pixels sit far above saturation 62 and background near 0, so
  MIN_TISSUE_FRACTION dominates and any cutoff in the 25–62 gap lands in the same place.
  Otsu's value is ADAPTIVITY: a faintly-stained slide gets a low cutoff automatically, where
  the fixed 25 would over-call background. Both masks were rendered over the slide overview
  and eyeballed — both cover all four tissue sections and reject background, dust, and pen.
  Verified: full e2e passes on Otsu (sweep -> complete, 48 tissue, train, predict, similarity,
  no GPU on the save path). Swapping SLIDEPROBE_TISSUE to hsv makes the existing otsu bank
  report state "none" (REJECTED, not silently reused) and swapping back accepts it again with
  no re-sweep. Degenerate check: on a background-only histogram raw Otsu returns 3 (splitting
  noise) -> clamped to 8, and a blank white tile is still correctly rejected.
  NOTE: changing the mask invalidates existing banks BY DESIGN — a re-sweep is required, which
  on Virchow 2 is minutes per slide. That is the right trade: mixing two masks' verdicts in
  one bank would be a silent correctness bug.

2026-07-14 — FREEHAND (lasso) drawing. Attempt #3, and this one works.
  THE PREVIOUS POST-MORTEM (above, 2 entries) WAS WRONG, and the wrong diagnosis is why two
  attempts died. It blamed "the pack's OSD support" and concluded a version bisect couldn't
  help. Reading the shipped bundles gives the real cause: the core's
  format(shape, annotation, formatters) takes an ARRAY, and @recogito/annotorious-selector-
  pack@0.6.1 vendored a copy of it but calls it with config.formatter -- the SINGULAR
  function. annotate.js sets `formatter: annotationFormatter` (to colour regions by class),

  formatter is configured -- which is exactly why the old isolated harness "proved" circle
  worked: it omitted the one config key that triggers the bug. The harness didn't lie because
  of flaky mouse-drags; it lied because it wasn't the real app. Bisecting the HOST library
  could never have fixed a bug in the PACK.

  THE FIX: don't add a shape type at all. A freehand stroke IS a polygon with many points,
  and polygons are NATIVE to Annotorious -- handled by the core's own EditablePolygon, which
  calls the correct plural API. So frontend/js/freehand.js captures the stroke itself (plain
  pointer events on a transparent capture layer that swallows them, so neither OSD nor
  Annotorious ever sees the drag and there is nothing to fight over), simplifies it
  (Ramer-Douglas-Peucker, ~2 screen px), and hands the points to SlideAnnotate.addPolygon().

  already parses an SvgSelector polygon, and _bbox/_point_in_polygon are generic over any

  * The SVG must be ONE LINE. Annotorious picks the editing tool with /^<svg.*<polygon/ --
    no `s` flag, so `.` doesn't cross newlines. Pretty-print it and the shape still renders
    but with NO vertex handles and no dragging, and nothing says why.
