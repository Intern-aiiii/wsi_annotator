SlideProbe change log
=====================

2026-07-06
- Added a "Change log" section to CLAUDE.md instructing that a running log be kept
  in docs/log.txt and updated after each fulfilled user request.
- Created this docs/log.txt file to start that history.
- Scaffolded the project directory skeleton from the proposed layout in CLAUDE.md:
    * backend/  — app.py, slides.py, patches.py, embeddings.py, classifier.py,
      inference.py, models.py (+ __init__.py). All are commented stubs describing
      each module's role and the phase it belongs to; no implementation yet.
    * frontend/ — index.html and js/ (viewer.js, annotate.js, api.js), stubbed.
    * data/     — slides/, cache/, annotations/, models/ with .gitkeep so the
      empty dirs exist while their contents stay gitignored.
    * requirements.txt — planned dependencies (FastAPI, OpenSlide, torch/timm,
      scikit-learn, etc.), versions mostly unpinned for now.
    * .gitignore — ignores data contents, Python cruft, and Zone.Identifier files.
- Implemented Roadmap Phase 1 ("One slide on screen"):
    * backend/slides.py — OpenSlide + DeepZoomGenerator: list slides, resolve a
      slide_id to a file (with a path-traversal guard), serve the .dzi descriptor
      and individual JPEG tiles; opened generators are cached per slide.
    * backend/app.py — FastAPI routes: GET /api/slides, GET /slides/{id}.dzi,
      GET /slides/{id}_files/{level}/{col}_{row}.jpeg; static frontend mounted at
      "/" (last, so API/tile routes win).
    * frontend — index.html loads OpenSeadragon (CDN) + a slide picker; js/api.js
      wraps the backend calls; js/viewer.js lists slides, fills the picker, and
      opens the first slide's DZI in OpenSeadragon.
    * Marked Phase 1 done in CLAUDE.md.
  Verified: Python byte-compiles; slide listing, id resolution, traversal guard,
  and tile-URL parsing pass a stubbed-openslide test. NOT run end-to-end here —
  the env has no FastAPI/OpenSlide and no sample WSI (see README to run it).
- Tested Phase 1 end-to-end with a real slide. Since the machine has no sudo,
  installed OpenSlide via the `openslide-bin` wheel (bundles libopenslide 4.0.1,
  no apt needed) into an isolated scratchpad venv, plus FastAPI/uvicorn/Pillow.
  Downloaded the Aperio CMU-1-Small-Region.svs (~1.9 MB, 2220x2967, MPP 0.499)
  into data/slides/ (gitignored). Results:
    * /api/slides lists the slide; /slides/<id>.dzi returns valid DZI XML
      (2220x2967, TileSize 254, Format jpeg).
    * Tiles decode as real JPEGs (magic ffd8ff); stitched a level-9 overview and
      visually confirmed H&E tissue renders correctly.
    * `uvicorn backend.app:app` boots and serves all routes over a real socket.
    * Error paths verified: 404 missing slide, 404 out-of-range tile, 400
      malformed tile name.
  Backend fully verified; the only unexercised piece is the OpenSeadragon render
  in an actual browser (standard DZI consumer — its inputs are confirmed valid).
  Kept the sample .svs in data/slides/ so the app can be run locally as-is.
- Reverted (at user request) the two changes made while answering "how do I run
  the app": removed the openslide-bin lines from requirements.txt and deleted the
  project-local .venv/. Repo is back to its post-Phase-1 state (sample .svs kept).
- Debugged "browser did not load". Root cause was NOT the code or WSL networking:
  a leftover uvicorn process (started from .venv) was stuck in D-state I/O sleep,
  still holding 127.0.0.1:8000 but never responding — because .venv had been
  deleted out from under the running process. Killed the stale process; verified
  on a clean server that GET / (200 html), /api/slides, /js/*, and tile serving
  all work. Confirmed it was purely the wedged process.
- Recreated the project .venv (at user's request) with the Phase-1 deps
  (fastapi, uvicorn, pillow, openslide-python, openslide-bin — no sudo needed)
  and verified the app boots and serves all routes. Left port 8000 free.
  Guidance to user: Ctrl+C the server before rebuilding .venv; check for stale
  listeners with `ss -ltnp | grep :8000` if the browser hangs again.

2026-07-07
- Implemented Roadmap Phase 2 ("Annotation"): draw regions on a slide, label
  each with a class, and persist per-slide. Per-annotation class labels are
  included now (not deferred to Phase 7) so saved annotations already carry the
  region+label shape Phases 3/5 need.
    * backend/annotations.py (new) — per-slide JSON store at
      data/annotations/<slide_id>.json. load()/save() plus annotations_path(),
      which reuses slides._resolve_slide_path so only real slides are writable
      (path-traversal guard). save() writes atomically (temp file + os.replace).
    * backend/models.py — added permissive AnnotationCollection Pydantic model
      (annotations: list[dict]); stores raw W3C JSON as-is.
    * backend/app.py — GET/PUT /api/slides/{id}/annotations (404 for unknown
      slide; PUT does a whole-collection replace, returns {"saved": N}).
    * frontend — Annotorious v2 (@recogito/annotorious-openseadragon@2.7.19) +
      toolbar plugin (@recogito/annotorious-toolbar@1.2.1) via jsDelivr CDN
      (versions pinned). viewer.js now exposes window.osdViewer and fires a
      "slideprobe:slide-opened" event on open; annotate.js attaches Annotorious,
      adds a TAG widget for the class label, mounts the draw-tool/mode toolbar,
      and saves the whole collection on create/update/delete. api.js gained
      get/saveAnnotations. index.html loads the CDN assets and hosts the toolbar
      + save-status UI in the topbar.
    * Marked Phase 2 done in CLAUDE.md.
  Verified: backend byte-compiles; against the sample CMU-1-Small-Region slide,
  the annotations API round-trips (empty GET -> PUT one W3C annotation ->
  {"saved":1} -> GET returns it -> file written to data/annotations/). 404s fire
  for unknown slides on both GET and PUT; a URL-encoded traversal id (..%2F..)
  never reaches the handler (fails route matching). Static frontend serves
  index.html + all js/, and the three pinned CDN assets return 200. NOT exercised
  here: the in-browser draw/label/render interaction (needs a real browser) —
  see the plan's manual step. Reset the test annotation file back to empty.
- Debugged "no Annotorious toolbar; UI looks like Phase 1". Server was serving
  the correct new assets; cause was browser-side (stale cache / slide rendering
  from OSD's earlier cache). Hardened annotate.js to surface init/plugin-load
  failures in the top-bar status instead of failing silently, and made the
  toolbar optional so drawing still works if only the toolbar plugin is missing.
  A hard refresh resolved it for the user.
- Expanded annotation class labels (user wanted more than gland/not-gland).
  Chose "both": a built-in PRESET list of common H&E categories (gland,
  epithelium, stroma, tumor, necrosis, lymphocytes, blood vessel, adipose,
  background) PLUS user-managed custom classes.
    * frontend/js/annotate.js — class list (presets + custom) drives the TAG
      widget vocabulary; custom classes persist in localStorage; a "Classes"
      panel adds/removes them; each class gets a stable palette color applied to
      its annotations via an Annotorious formatter + an injected stylesheet;
      typing a brand-new label remembers it as a custom class automatically.
      Changing the class list rebuilds the Annotorious instance (destroy+
      recreate, preserving displayed annotations) to refresh the vocabulary.
    * frontend/index.html — added the Classes button + floating panel markup and
      styling. Class labels are still stored as standard W3C tagging bodies, so
      no backend change was needed.
- Attempted extra drawing tools (circle, freehand, point) via the official
  @recogito/annotorious-selector-pack@0.6.1, then REVERTED at user request.
    Why reverted: the tools drew shapes but did not integrate with the editor —
    the new shapes could not be tagged or deleted like rect/polygon. Diagnosed
    with a headless-Chromium (Playwright) end-to-end test driving the actual
    tools: the shape is created but the draw-complete -> open-editor path throws
    ("n.reduce is not a function", "getAttribute is not a function", "reading
    'element' of null"), so the editor never opens (hence no tag / no delete).
    Bisected annotorious-openseadragon 2.6.0 through 2.7.19 against the pack:
    circle actually worked (editor opened, no errors) at ALL versions, but
    FREEHAND failed at every version and POINT is documented as OSD-incompatible
    (removed from the pack for OSD). So the breakage is inherent to the pack's
    OSD support, not a version mismatch we can pin around.
    Reverted the index.html selector-pack script and the annotate.js
    SelectorPack() call; toolbar is back to rect + polygon only. (Debugging-only
    Playwright install was also removed from the venv.)
    Salvage note for later: circle looked reliable in an ISOLATED harness, but
    NOT in the real app.
- Tried again to add circle-only (user request), then REVERTED again. In the
  full app (toolbar + per-class formatter), drawing a circle failed the same way
  as freehand: the editor did not open and the page threw "n.reduce is not a
  function" / "reading 'element' of null", so it couldn't be tagged or deleted.
  The isolated harness that had made circle look OK omitted the formatter/toolbar
  and its headless mouse-drag simulation proved flaky (circle passed once, failed
  on reruns), so that earlier "circle works" result was not trustworthy. Net:
  none of the selector-pack tools (circle/freehand/point) integrate reliably with
  the editor on this Annotorious OSD setup. Reverted the index.html script and the
  annotate.js SelectorPack() call; toolbar is rect + polygon only. Removed the
  debugging Playwright install again. If curved/round regions are needed later,
  the realistic path is a maintained library (e.g. Annotorious v3) or a custom
  tool — not this pack.
